Higress plugins run as WebAssembly (Wasm) modules inside Envoy's sandbox. They extend gateway behavior without restarts or traffic drops. Plugins are loaded via the `WasmPlugin` CRD and can be applied globally, per domain, or per route.

## Supported languages

| Language | SDK | Notes |
|---|---|---|
| **Go** | `github.com/higress-group/wasm-go` | Primary SDK; most official plugins use Go |
| **Rust** | `higress-wasm-rust` | Full-featured SDK with rule matching and Redis support |
| **JavaScript / AssemblyScript** | proxy-wasm-as | Lighter alternative |
| **C++** | proxy-wasm-cpp-sdk | Used by some legacy built-in plugins |

## Plugin execution phases

Each request and response passes through four phases. Your plugin registers handlers for the phases it needs.

| Phase | Hook (Go) | Hook (Rust) |
|---|---|---|
| Request headers received | `ProcessRequestHeaders` | `on_http_request_headers` |
| Request body received | `ProcessRequestBody` | `on_http_request_body` |
| Response headers received | `ProcessResponseHeaders` | `on_http_response_headers` |
| Response body received | `ProcessResponseBody` | `on_http_response_body` |

Streaming response bodies can be intercepted chunk by chunk using `ProcessStreamingResponseBody` (Go) without buffering the full response.

## Plugin scopes

A single `WasmPlugin` resource can apply different configurations at different scopes. Rules are evaluated in order; the first match wins.

| Scope | `matchRules` key | Example |
|---|---|---|
| Global (default) | `defaultConfig` | Applied to all requests not matched by a rule |
| Route (Ingress) | `ingress` | `default/my-ingress` |
| Domain | `domain` | `*.example.com` |

```yaml
apiVersion: extensions.higress.io/v1alpha1
kind: WasmPlugin
metadata:
  name: request-block
  namespace: higress-system
spec:
  defaultConfig:
    block_urls:
    - "swagger.html"
  matchRules:
  - ingress:
    - default/foo
    config:
      block_urls:
      - "admin"
  - domain:
    - "*.internal.example.com"
    config:
      block_urls:
      - "debug"
  url: oci://higress-registry.cn-hangzhou.cr.aliyuncs.com/plugins/request-block:1.0.0
```

## WasmPlugin CRD

The `WasmPlugin` CRD (API version `extensions.higress.io/v1alpha1`) is the primary way to load a plugin. Key fields:

| Field | Description |
|---|---|
| `spec.url` | OCI image URL or local `file://` path to the `.wasm` binary |
| `spec.defaultConfig` | Plugin configuration applied globally |
| `spec.matchRules` | List of scope-specific configuration overrides |
| `spec.selector` | Label selector to target specific gateway pods |

```yaml
apiVersion: extensions.higress.io/v1alpha1
kind: WasmPlugin
metadata:
  name: key-auth
  namespace: higress-system
spec:
  selector:
    matchLabels:
      higress: higress-system-higress-gateway
  defaultConfig:
    consumers:
    - credential: my-api-key
      name: consumer-a
  url: oci://higress-registry.cn-hangzhou.cr.aliyuncs.com/plugins/key-auth:1.0.0
```

## Deploying a plugin

<Steps>
  <Step title="Build or pull the plugin image">
    Use the official registry or build your own. Official images follow the pattern:

    ```
    oci://higress-registry.cn-hangzhou.cr.aliyuncs.com/plugins/<name>:<version>
    ```
  </Step>
  <Step title="Apply the WasmPlugin resource">
    ```bash
    kubectl apply -f my-plugin.yaml
    ```

    Configuration changes take effect within milliseconds with no connection loss.
  </Step>
  <Step title="Verify the plugin is loaded">
    ```bash
    kubectl get wasmplugin -n higress-system
    kubectl describe wasmplugin my-plugin -n higress-system
    ```
  </Step>
</Steps>

<Note>
All official plugin images are hosted at `higress-registry.cn-hangzhou.cr.aliyuncs.com/plugins/`. You can also push your own images to any OCI-compatible registry.
</Note>
